Incident Intelligence System Overview
Overview

What is the Incident Platform?

An operational command dashboard built for IT Support Groups, NOC Engineers, and Incident Managers.

Operational groups are constantly bombarded with tickets. Our platform leverages historical patterns and real-time AI to minimize operational delay, optimize routing, track commitments, and fast-track resolving procedures.

Core Business Goals

  • Zero Manual Routing Delay: Pre-routes tickets instantly to correct support teams.
  • Dynamic SLA Compliance: Adjusts commitments based on holds/third-party delays.
  • Automated Post-Mortem Action: Compiles Root Cause Analysis (RCA) summaries in one click.
1. Automated Triage

Predicts Routing Team, Severity, and Expected Resolution Time immediately upon creation.

2. Intercept Duplicate Outages

Detects concurrent duplicate issues with natural text similarities and flags them to the operator.

3. Cognitive Root Cause & Escalation

Provides AI-reasoned root causes, check steps, and checks SLA breach likelihood to recommend L3 escalation.

State Machine

Incident Status Lifecycle

The platform enforces a strict IT Service Management (ITSM) state transition workflow. This prevents ticket state errors and guarantees audited timestamp tracking at every phase of resolution.

Lifecycle Transitions & Rules

  • Open → Assigned / Cancelled: Fresh tickets wait for assignment.
  • Assigned → In Progress / Cancelled: Operator acknowledges and initiates investigations.
  • In Progress → Resolved: Problem solved; logs actual resolution time and determines final SLA metrics.
  • Resolved → Closed: Terminal state. Ticket completed and locked.

* No layout shifts occur in the dashboard UI when status transitions update.

Architecture

The Incident Pipeline

Data Processing Pipeline

  1. Input Ingestion: Incident recorded with summary, affected scope, target environment, and user volume.
  2. Synonym Normalization & Check: Preprocesses ticket content using regex arrays to match acronyms and ticket synonyms (e.g. log in vs login). Checks TF Cosine similarities against active tickets.
  3. Submission Intercept: If score ≥ 80%, submission pauses to prompt operator (Option to view existing match, abort, or save anyway).
  4. AI Predictions & Storage: Runs description and parameters against ML models (team, priority, MTTR), fallbacks to rules on storage errors, and inserts into PostgreSQL.
Core Capability

Major Operations Features

⏸ SLA Live Timer

Standardizes incident response commitments dynamically.

  • P1 Severity Response threshold: 1 Hour.
  • Allows operators to pause target SLA timer when awaiting external inputs.
  • Pause metrics automatically recorded and adjust SLA deadlines.
🧠 AI Root Cause Agent

Generates detailed root cause findings using historical knowledge base.

  • Compares symptoms against 5 most similar past issues.
  • Generates root cause hypothesis, confidence scores, and troubleshooting steps.
  • Supports manual overrides and audits of root causes.
⚡ L3 Escalation Advisor

Determines incident complexity and escalation thresholds.

  • Determines risk percentages (0-100%) from severity & outage data.
  • Generates escalation justifications based on historical resolutions.
  • Provides recommended support group and teams target.
Operational Ease

Minor & Auxiliary Features

✓ System Orchestration & Integration
  • Sequential Year-Based IDs: INC-YYYY-000XX format (resets dynamically at calendar transitions; ignores debug codes).
  • Multi-Team Intersection Filters: Supports assigning multiple comma-separated teams (e.g. Network, Database) and filters tickets using intersection matching.
  • No Layout Shifts: Elements (timeline steps) are locked to a fixed 24px height; layout placeholders keep UI elements static when values load async.
☁ Data Export & Cloud Storage
  • Secure PDF Exports: In one click, compiles detailed RCA reports containing incident metrics, summaries, and actions to PDF binaries via ReportLab.
  • Cloud Storage Uploader: Automatically pushes generated PDF files to Azure Blob Storage containers (`rca-files`).
  • Secure Downloading: Streamlit leverages Azure storage clients to pull report binaries directly from cloud blobs securely.
Data Model

Database Schema & Columns

📁 Core Operational Columns
  • incident_id (PK): Unique sequential ID (e.g. INC-2026-00001).
  • description: Detailed symptom description of the outage.
  • application / environment: Target service and sandbox info.
  • affected_users / impact_scope: Outage volume indicators.
  • category: Derived category mapped dynamically.
  • assigned_team / priority: Mutable groups and priority keys.
  • status: ITSM state machine lifecycle marker.
  • actual_resolution_time: Duration mapped upon ticket resolution.
⚙ AI Predictions & Audit Logs
  • ai_predicted_...: Original ML predictions (team, priority, time).
  • ..._overridden: Flag markers indicating custom manual inputs.
  • sla_breached: Flag checking if MTTR exceeded SLA limits.
  • sla_pause_log: JSON audit array tracking holds and resumes.
  • l3_escalation_...: L3 risk score, recommended flags, and reasons list.
  • rca_...: Post-mortem generated content and URL paths.

Roadmap

Scope for Future Improvements

1 Offline Core AI Models

Reduce external dependencies for sensitive data:

  • Host local Small Language Models (e.g. Llama-3-8B-Instruct or Mistral-7B) inside private Kubernetes clusters.
  • Eliminates dependency on external APIs (OpenRouter/Gemini), lowering latency and improving compliance.
2 Advanced Outage Analysis

Enhanced batch schedule and log analysis:

  • Correlate batch job dependency trees (Control-M/Autosys paths) directly during duplicate incident warnings.
  • Add RAG (Retrieval-Augmented Generation) query pipelines on system logs and deployment release notes.
3 Orchestration & Workflow Policies

Multi-Factor SLA Pause Approvals: Require secondary reviews or operations lead authorization when a technician keeps the SLA clock paused for longer than 60 minutes.

Smart Workload Load Balancing: Dynamically suggest ticket shifts to other engineers if a particular technician's active priority workload becomes elevated.
