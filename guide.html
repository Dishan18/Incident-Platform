Overview & Architecture

Incident Management Triage & Decision Support Command Center Guide

System Overview

The Incident Intelligence Platform is a dual-tier Machine Learning and Generative AI platform designed to assist NOC, Infrastructure, Database, and Support teams. The system analyzes ticket description symptoms to automatically predict assignment groups, estimate priority severity levels, calculate expected resolution durations, retrieve similar historical resolutions, and generate root cause analyses.

Recent enhancements introduce active SLA tracking (with pause/resume states for In Progress tickets), L3 escalation risk assessment using OpenRouter, and a duplicate ticket detection engine that matches text similarity on submission.

The platform is designed around a relational SQLite database layer supporting state transitions (Open to Assigned to In Progress to Resolved to Closed or Cancelled) and interactive dashboard views.

Architecture Overview

1. Live Log Ingestion

Incidents are created via a form modal, invoking predictions and similarity checks immediately. An incident record gets initialized in the SQLite database.

2. Static ML Prediction

Random Forest classifiers run feature extraction to predict support groups and priorities. A regression model estimates resolution time.

3. Cognitive AI Routing

TF-IDF cosine similarity retrieves similar historical resolutions. Generative agents run root cause and L3 escalation analysis using OpenRouter.

Core Objective

To accelerate Incident Lifecycle Routing and reduce Mean Time To Resolution (MTTR) by eliminating manual routing errors and providing immediate, explainable AI diagnoses to NOC engineers.

ML Telemetry & Models

  • Team Classifier: Random Forest pipeline trained on TF-IDF word tokens from historical descriptions to classify assignment groups.
  • Priority Classifier: Random Forest classifier combining description vectors, application categories, environment values, and impact scales.
  • Resolution Estimator: Ridge regression model mapping features to numerical metrics representing resolution duration in minutes.
  • L3 Escalation Advisor Heuristics: Dynamic fallback checks that evaluate priorities (P1/P2), scale of affected users, scope of business impact, and estimated resolution complexity when API integration is inactive.

How to Run the Project

Follow these steps to configure your local system and start the Incident Intelligence Platform:

1. Set Up Environment Variables

Create a .env file in the root workspace folder and declare your API keys:

# Google Gemini API key configuration
GEMINI_API_KEY=your_gemini_api_key_here

# OpenRouter API key configuration
OPENROUTER_API_KEY=your_openrouter_api_key_here

# Server port configuration
PORT=8501

2. Install Dependencies

Activate your virtual environment and install the required modules:

# Create and activate environment (if not already done)
python -m venv venv
.\venv\Scripts\activate

# Install required packages
pip install -r requirements.txt

3. Initialize SQLite Database

Set up your relational database tables using one of the following methods:

A. Fresh Database Initializations

Compile database schemas and migrate legacy datasets:

# Create database tables and initialize the schemas
python -m scripts.init_db

# Migrate historical live ticket samples from CSV into SQLite db
python -m scripts.migrate_csv_to_db

B. Migrating Existing Database Instances

If you have an existing database, run the following scripts to add columns for SLA tracking and L3 Escalation Advisory:

# Add SLA breached status column
python -m scripts.add_sla_breached_column

# Add SLA pause logs column
python -m scripts.add_sla_pause_log_column

# Add L3 Escalation columns
python -m scripts.add_l3_escalation_columns

4. Run Testing Verification Scripts (Optional)

Validate the operations, calculations, and AI responses via CLI:

# Test SLA calculations and pausing rules
python -m backend.incident.test_sla

# Test L3 Escalation Advisor heuristics and LLM parser
python -m backend.ml.test_l3_escalation

# Test Duplicate Incident checks
python -m backend.ml.test_duplicate_detection

5. Start Streamlit Command Center

Launch the dashboard application web server:

streamlit run app.py --server.port 8501 --server.address 127.0.0.1
Deployment URL

Open http://127.0.0.1:8501 in your web browser to access the dashboard command center.

Containerization & Kubernetes Deployment

The platform supports full containerization using Docker and deployment orchestration via Kubernetes. This enables reliable hosting, horizontal scaling, and isolated runtime environments matching enterprise ITSM workflows.

How Containerization and Orchestration Work

  • Docker (Dockerfile): Uses a lightweight python:3.11-slim base image, copies dependencies in requirements.txt, and runs pip installation with caching disabled. Copies the application and exposes port 8501 (the default port for Streamlit web apps).
  • Kubernetes Deployment (k8s/deployment.yaml): Manages the lifecycle of Streamlit pods, defines a single replica pod running ticketing-platform:v1, and exposes port 8501 inside the pod.
  • Kubernetes Service (k8s/service.yaml): Configures a NodePort service routing incoming cluster node requests on port 8501 directly to container port 8501, exposing the platform across the local system.
Development Convenience

By exposing Streamlit through a service NodePort or setting up a port-forward, NOC engineers can load and access the AI-powered dashboards securely on their local workstations or across internal subnets.

Steps to Execute

A. Docker Execution Steps

To run the platform locally using Docker, execute the following commands in the project root directory:

  1. Build the Container Image: Build the container image tagging it as ticketing-platform:v1. 
    docker build -t ticketing-platform:v1 .
  2. Run the Local Container: Run the container locally by mapping port 8501 and mounting your .env file containing your OpenRouter API keys. 
    docker run -p 8501:8501 --env-file .env ticketing-platform:v1
  3. Access the Dashboard: Once started, access the UI command center in your browser at http://localhost:8501.

B. Kubernetes Execution Steps

To run the platform inside a local Kubernetes cluster (such as Minikube or Kind):

  1. Push/Load Image to Cluster: Ensure your Kubernetes cluster context has access to the local Docker image: 
    # For Minikube:
minikube image load ticketing-platform:v1

# For Kind:
kind load docker-image ticketing-platform:v1
  2. Apply Configuration Manifests: Apply the deployment and service specifications: 
    kubectl apply -f k8s/deployment.yaml
kubectl apply -f k8s/service.yaml
  3. Verify Running Resources: Verify that the deployment pod is successfully scheduled and the NodePort service is created: 
    kubectl get deployments,pods,services
  4. Port-Forward service: Forward traffic from port 8501 on your local computer to the service exposed in the cluster: 
    kubectl port-forward service/ticketing-platform-service 8501:8501
  5. Access the Platform: Open http://127.0.0.1:8501 to access the containerized Streamlit command center dashboard.

Future Roadmap & Enhancements

Below are primary architectural improvements designed to scale the system for enterprise-grade workloads:

Sentence-Transformers PostgreSQL Integration Multi-Agent NOC Orchestration API Caching

1. Semantic Search via Sentence-Transformers

Current Limitation: The historical ticket similarity search currently runs on TF-IDF vectorization. TF-IDF relies on exact keyword frequencies, meaning it cannot associate synonyms (e.g. associating "database connection timeout" with "DB server unreachable") or correct typos.

Proposed Solution: Transition similarity calculation to use Sentence-Transformers (like all-MiniLM-L6-v2 or bge-small-en-v1.5) to generate dense semantic embeddings. 

# Example snippet for Sentence-Transformer implementation
from sentence_transformers import SentenceTransformer
import numpy as np

# Load pre-trained embedding model
model = SentenceTransformer('all-MiniLM-L6-v2')

# Pre-compute and save historical embeddings
historical_embeddings = model.encode(historical_descriptions)

# Retrieve top matches using cosine similarity
query_embedding = model.encode([current_description])
similarities = cosine_similarity(query_embedding, historical_embeddings)[0]
top_indices = np.argsort(similarities)[::-1][:5]

This allows the retrieval layer to perform true semantic matchmaking, returning relevant historical occurrences regardless of terminology shifts.

2. SQLite to PostgreSQL Migration

To support high-concurrency logging, clustering, and audit trails across hundreds of concurrent NOC operators, migrate database connections from local SQLite files to an enterprise-grade PostgreSQL instances. Our SQLAlchemy models are already abstracted and fully compatible with Postgres dialects.

3. Automated Remediation Hooks

Integrate the generative steps output by the Gemini agent into automated playbook execution APIs (such as Ansible, RunDeck, or Kubernetes operators). On high confidence scores (e.g., > 90%), the system can trigger automated diagnostics, system reboots, or patch deployments, transforming the platform from a decision support tool into an active autonomous NOC agent.

4. Redis Cache Integration

Cache ML predictions and Gemini Root Cause responses in a Redis cluster to guarantee sub-millisecond response times for repeating alerts, minimizing LLM request costs and API quota exhaustion risk.
